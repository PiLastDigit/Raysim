# Phase B2 — Materials + Project File Implementation Plan

**Version target**: `0.2.0` (minor bump from `0.1.0` — adds the `raysim.mat`
package and `raysim.proj.project` module).
**Phase reference**: `MVP_PLAN.md §6 Phase B2`, `MVP_STEPS.md §B2.1–§B2.6`.

## Overview

Phase B2 builds the material governance layer so that every solid in the
assembly resolves to a library material before a run. It fills the
`raysim.mat` placeholder with a seeded material library, a naming-rule
auto-assignment engine, an OCCT-optional STEP tag ingestion module, and an
auto-assignment review API. It also introduces the `.raysim` project file
format — canonical JSON, bit-stable across save/reopen/save, carrying
geometry references, material assignments, detector/scenario state, and
naming-rule overrides.

## Problem Statement

Stage A accepts materials as a flat CSV/YAML plus an optional JSON
assignments file, both supplied as CLI flags. This works for headless
regression testing but not for the Stage B product workflow where a user
opens a STEP, assigns materials via a library, and expects assignments to
persist in a project file. The material library is hardcoded in test
fixtures; there is no seeded library, no STEP-tag ingestion, no naming-rule
engine, and no project file format. Phase B2 fills all of these gaps.

## Solution Architecture

Three new modules inside `raysim.mat/` (library, rules, step_tags) plus one
new module in `raysim.proj/` (project). All pure-Python except
`mat.step_tags`, which imports `pythonocc-core` at call time and is guarded
by `importorskip` in tests. The `.raysim` project file is a Pydantic model
serialized via the existing `canonical_json` module — bit-identical on
save/reopen/save. Run gating and density anomaly warnings live in `mat/`
as pure functions callable from both CLI and future UI.

Layering:

```
raysim.mat.library     ← seeded YAML, user extensions
raysim.mat.rules       ← naming-rule engine (regex → group_id)
raysim.mat.step_tags   ← OCCT XCAFDoc_MaterialTool (optional)
raysim.mat.review      ← auto-assignment review API + CLI summary
raysim.mat.gating      ← run-readiness check + density anomaly warnings
raysim.proj.project    ← .raysim project file schema + I/O
```

The existing `proj.schema.Material` and `proj.schema.MaterialAssignment`
models remain unchanged and are the canonical types consumed downstream.
`raysim.mat` produces instances of these types; it does not define its own
parallel hierarchy.

## Implementation Details

### 1. Material Library (`raysim.mat.library`)
**File**: `src/raysim/mat/library.py` (new)

Loads material entries from a YAML library file. Ships a default seeded
library at `src/raysim/mat/default_library.yaml`.

**Responsibilities**:
- `load_library(path: Path | None = None) -> MaterialLibrary`: loads the
  YAML, falls back to the bundled default if `path` is None.
- `MaterialLibrary`: a frozen dataclass wrapping `tuple[Material, ...]` with
  a `by_group_id: dict[str, Material]` lookup. Enforces uniqueness of
  `group_id`. Provides `merge(other: MaterialLibrary) -> MaterialLibrary`
  for user extensions.
- The default YAML contains 14 entries per `MVP_STEPS.md §B2.1`:
  11 base materials (Al 6061, Cu, Si, SiO₂, Kapton, FR4, Ti-6Al-4V, W, GaAs,
  Au, Sn-Pb solder) plus 3 composites (battery, populated PCB, harness).
  Each entry has a NIST or vendor-datasheet-sourced `density_g_cm3`, optional
  `z_eff`, `display_name`, and `provenance` citing the source.

**YAML schema** (per entry — maps 1:1 to `proj.schema.Material`):
```yaml
materials:
  - group_id: aluminum_6061
    density_g_cm3: 2.70
    z_eff: 13.0
    display_name: Al 6061
    provenance: "NIST SRD 124, density for 6061-T6"
  - group_id: copper
    density_g_cm3: 8.96
    z_eff: 29.0
    display_name: Cu (OFHC)
    provenance: "NIST"
  # ... 11 more entries
```

### 2. Default Library YAML
**File**: `src/raysim/mat/default_library.yaml` (new)

The seeded library shipped with the app. 14 entries per `MVP_STEPS.md §B2.1`.
Each entry is a dict matching the `Material` Pydantic schema. Library

Must be included in the package distribution (add to `pyproject.toml`

### 3. Naming-Rule Engine (`raysim.mat.rules`)
**File**: `src/raysim/mat/rules.py` (new)

Regex-pattern-to-`group_id` matching. Ships a default ruleset at
`src/raysim/mat/default_rules.yaml`.

**Responsibilities**:
- `NamingRule`: frozen dataclass with `pattern: str` (regex), `group_id: str`,
  `priority: int` (higher wins on tie), `source: str` (e.g. "default",
  "project").
- `load_rules(path: Path | None = None) -> tuple[NamingRule, ...]`: loads
  YAML, falls back to bundled default.
- `SolidRef`: frozen dataclass with `solid_id: str`, `path_key: str`,
  `display_name: str` (defaults to `solid_id`). The stable input token that
  rules and review APIs operate on. B1's `LeafSolid` maps 1:1 to a
  `SolidRef`; Stage A STL-only callers build `SolidRef(solid_id=stem, path_key=stem)`.
- `apply_rules(rules, solids: Sequence[SolidRef]) -> list[RuleMatch]`: for
  each solid, evaluates all rules against both `solid_id` and `path_key`.
  Returns a `RuleMatch` per solid: `solid_id`, `matched_group_id | None`,
  `is_ambiguous: bool`, `candidates: tuple[NamingRule, ...]` (all rules that
  matched). Ambiguous = 2+ rules at the same priority both match. Pattern is
  matched against `solid_id`, `path_key`, and `display_name`; first match
  wins per priority.

**Default rules YAML** covers `MVP_STEPS.md §B2.3` conventions:
```yaml
rules:
  - pattern: "(?i)(^|[_/])al(u?m?i?n?[ui]?[um]?m?|_?6061)([_/]|$)"
    group_id: aluminum_6061
    priority: 10
  - pattern: "(?i)(^|[_/])cu(pper)?([_/]|$)"
    group_id: copper
    priority: 10
  # ... patterns for SI, FR4, KAPTON, TI, W, AU, GAAS, PCB, BATT
```

### 4. STEP Material-Tag Ingestion (`raysim.mat.step_tags`)
**File**: `src/raysim/mat/step_tags.py` (new)

OCCT-dependent. Imports `pythonocc-core` at call time (not at module level).
Tests guarded by `pytest.importorskip("OCC.Core")`.

**Responsibilities**:
- `extract_step_tags(step_path: Path, leaves: Sequence[LeafSolid]) -> list[StepMaterialTag]`:
  uses `STEPCAFControl_Reader` + `XCAFDoc_MaterialTool` + `XCAFDoc_ColorTool`
  to extract per-shape material name strings and color attributes.

  **Correlation strategy** (addresses the two-reader problem): B1's
  `step_loader` uses a plain `STEPControl_Reader`, while tag extraction
  requires the XCAF reader. Shape object identity does not survive across
  the two independent readers. The join key is **leaf solid index in DFS
  walk order**: both B1's `_build_tree` and the XCAF label-tree traversal
  walk the compound in the same depth-first order dictated by OCCT's
  `TopoDS_Iterator`. The implementation:
  1. Walks the XCAF label tree in DFS order, collecting one material-name
     + color per leaf label, plus the leaf's bounding box.
  2. **Verification gate (two checks)**:
     a. **Count check**: asserts the XCAF walk produces the same leaf count
        as `len(leaves)`. On mismatch → logs a warning and returns an
        empty tag list.
     b. **Bbox fingerprint check**: for each index `i`, compares the XCAF
        leaf's bounding box against `leaves[i].bbox_min_mm` /
        `leaves[i].bbox_max_mm` within a 1e-3 mm tolerance. If any leaf
        pair diverges → logs a warning with the mismatched indices and
        returns an empty tag list.
     Both checks must pass before tags are zipped. This catches the
     "same count, different order" case that a count-only check misses.
  3. On verification pass, zips the XCAF leaf list with `leaves` by index,
     keying each tag to `leaves[i].solid_id`.

  This is the simplest correct strategy that avoids modifying B1's loader.
  If a future STEP file triggers the verification-gate fallback, the fix
  path is to migrate B1 to use `STEPCAFControl_Reader` as its primary
  reader (producing both shapes and XCAF labels in a single pass),
  documented as a deferred improvement.

  Returns `StepMaterialTag(solid_id: str, material_name: str | None, color_rgb: tuple[float,float,float] | None)`.
- `match_tags_to_library(tags, library: MaterialLibrary) -> list[TagMatch]`:
  fuzzy-matches STEP material strings to library `group_id` / `display_name`.
  `TagMatch(solid_id: str, matched_group_id: str | None, confidence: float, raw_tag: str)`.
  Confidence thresholds configurable; default 0.7.

This module is the only OCCT-dependent piece in all of B2. The rest of the
`mat/` package and the project file work without pythonocc.

### 5. Auto-Assignment Review API (`raysim.mat.review`)
**File**: `src/raysim/mat/review.py` (new)

Combines the outputs of STEP tags, naming rules, and any pre-existing manual
assignments into a single per-solid assignment status.

**Responsibilities**:
- `AssignmentStatus`: frozen dataclass per solid: `solid_id`, `source`
  (Literal["step_tag", "naming_rule", "manual", "unassigned"]),
  `material_group_id: str | None`, `is_ambiguous: bool`,
  `candidates: tuple[str, ...]`.
- `build_review(solids: Sequence[SolidRef], *, tag_matches, rule_matches, manual_assignments, library) -> AssignmentReview`:
  priority order is `manual > step_tag > naming_rule`. An `AssignmentReview`
  carries: `statuses: tuple[AssignmentStatus, ...]`, summary counters
  (`n_auto_matched`, `n_ambiguous`, `n_unassigned`).
- `review_to_assignments(review: AssignmentReview) -> list[MaterialAssignment]`:
  converts resolved statuses into the canonical `MaterialAssignment[]` that
  the engine consumes. Raises `ValueError` if any solid is unassigned (run
  gating catches this too, but this is the explicit conversion-time check).
- `format_review_summary(review: AssignmentReview) -> str`: human-readable
  summary for CLI output (counts + per-solid table for ambiguous/unassigned).

Phase B3 will add a visual panel on top of `AssignmentReview`; B2 provides
the data model and a CLI-printable summary.

### 6. Run Gating and Density Anomaly Warnings (`raysim.mat.gating`)
**File**: `src/raysim/mat/gating.py` (new)

Pure functions that check whether a run is permitted.

**Responsibilities**:
- `check_run_readiness(assignments: Sequence[MaterialAssignment], solids: Sequence[str], library: MaterialLibrary) -> GatingResult`:
  verifies every `solid_id` in `solids` has an assignment that resolves to a
  library entry. Returns `GatingResult(ready: bool, missing_solids: tuple[str, ...], unresolved_assignments: tuple[str, ...])`.
- `check_density_anomalies(materials: Sequence[Material]) -> list[DensityAnomaly]`:
  flags entries with `density_g_cm3 < 0.5` or `> 25.0`. Returns
  `DensityAnomaly(group_id: str, density_g_cm3: float, kind: Literal["low", "high"])`.
  These are warnings, not blockers — per `MVP_PLAN.md §4.8`.
- `DensityAnomaly` and `GatingResult` are frozen dataclasses.

### 7. Project File Schema (`raysim.proj.project`)
**File**: `src/raysim/proj/project.py` (new)

The `.raysim` project file — Pydantic model, canonical JSON, bit-stable on
save/reopen/save.

**Responsibilities**:
- `PROJECT_SCHEMA_VERSION = 1`: independent version from `SCHEMA_VERSION` in
  `proj.schema` (the run-output schema). Bumped on breaking project-file
  changes. Loader accepts N−1.

- `GeometryRef`: frozen model with `path: str` (relative to project file),
  `sha256: str`, `tessellation_linear_mm: float`, `tessellation_angular_rad: float`.

- `ProjectFile`: frozen Pydantic model carrying:
  - `project_schema_version: int`
  - `geometry: GeometryRef`
  - `materials_library_path: str | None` (relative; None = use default)
  - `materials_library_sha256: str | None`
  - `assignments: tuple[MaterialAssignment, ...]`
  - `assignment_sources: dict[str, str]` (solid_id → source label for
    traceability: "step_tag", "naming_rule", "manual")
  - `naming_rule_overrides: tuple[NamingRuleOverride, ...] | None`
  - `detectors: tuple[Detector, ...]`
  - `dose_curve_path: str | None` (relative)
  - `dose_curve_sha256: str | None`
  - `interference_overrides: dict[str, str]` (pair key → justification text,
    carried from B1 `ValidatedAssembly.overrides_used` into the project)
  - `created_at_utc: str` (ISO 8601; not part of canonical hash)
  - `raysim_version: str`

- `NamingRuleOverride`: `pattern: str`, `group_id: str`, `priority: int`.
  Stored in the project to accumulate team conventions per `MVP_STEPS.md §B2.3`.

- `save_project(project: ProjectFile, path: Path) -> None`: writes canonical
  JSON via `raysim.proj.canonical_json.dumps()`.

- `load_project(path: Path) -> ProjectFile`: reads JSON, validates via
  Pydantic, checks `project_schema_version` (accepts current and N−1 with
  migration). On load, verifies `geometry.sha256` against the referenced file;
  warns (does not fail) if stale.

- `geometry_hash(project: ProjectFile, base_dir: Path) -> str`: computes
  SHA-256 of the geometry file referenced by `GeometryRef.path` relative to
  `base_dir`. Used by the run driver for provenance.

**Bit-stability contract**: `save(load(save(project))) == save(project)` as
bytes. Enforced by canonical JSON (sorted keys, `%.17g` floats). The
`created_at_utc` field is set once at project creation and **preserved
verbatim** on every subsequent `load_project` → `save_project` cycle — it is
never regenerated. This ensures bit-stability. It is also *not* part of any
provenance hash — same pattern as `human_metadata` in `run.json`.

**Hash clarity**: the project file contains several hashes serving different
purposes:
- `geometry.sha256`: SHA-256 of the referenced STEP/STL file bytes. Verified
  on load to detect stale references.
- `materials_library_sha256`: SHA-256 of the custom library YAML (None when
  using the default). Detects library drift.
- `dose_curve_sha256`: SHA-256 of the `.dos` file bytes.
- These per-artifact hashes feed into the engine's `Provenance` at run time
  (via `geometry_hash`, `materials_hash`, `assignments_hash`, etc. in
  `proj.schema.Provenance`). The project file itself has no single
  "project hash" — each input is hashed independently for provenance.

## Technical Considerations

- **Pattern Usage**: follows the existing `frozen=True` + `extra="forbid"`
  Pydantic convention (ARCHI §5). All new types are frozen dataclasses or
  frozen Pydantic models.

- **Determinism Impact**: the `.raysim` project file uses canonical JSON and
  is bit-stable on save/reopen/save. The `assignments` tuple produced by
  `review_to_assignments()` feeds into the existing `assignments_hash` in
  `Provenance` — no change to the run-output determinism contract.
  `created_at_utc` is excluded from provenance hashes.
  No `SCHEMA_VERSION` bump in `proj.schema` — the run-output schema is
  unchanged. The new `PROJECT_SCHEMA_VERSION` is independent.

- **Numerical Precision**: no impact. B2 does not touch the ray engine, dose
  math, or float accumulation paths.

- **Material Physics Scope**: strictly `density_g_cm3` only for dose math.
  The seeded library carries `z_eff`, `composition`, `display_name`, and
  `provenance` as metadata per `MVP_PLAN.md §3` — preserved for traceability,
  not consumed.

- **Coincident-Face / Tied-Batch**: no impact. B2 does not modify
  `raysim.ray.scene` or `raysim.ray.tracer`.

- **Backend Optionality**: `raysim.mat.step_tags` is the only OCCT-dependent
  module. It imports `OCC.Core` at call time, not at module level. Tests use
  `pytest.importorskip("OCC.Core")`. All other B2 modules are pure Python
  and fully testable in the existing `uv`-based CI.

- **Schema Versioning**: no bump to `SCHEMA_VERSION` (run-output schema
  unchanged). New `PROJECT_SCHEMA_VERSION = 1` for the `.raysim` file.
  Loader will accept N−1 once a v2 is introduced.

- **Edge Cases**:
  - Empty STEP with no solids → `build_review` returns all-unassigned,
    gating blocks the run.
  - Library with duplicate `group_id` → `MaterialLibrary` raises on
    construction.
  - Naming rule matches 0 solids → no error; `n_auto_matched = 0`.
  - Ambiguous rules (2+ at same priority match same solid) → solid marked
    ambiguous, not auto-resolved. User must manually assign.
  - Density anomaly at boundary (exactly 0.5 or 25.0) → not flagged
    (strictly less / strictly greater).
  - No STEP tags available (STEP has no material attributes) → tag extraction
    returns empty list; naming rules and manual assignment still apply.
  - Project file referencing a moved/renamed STEP → `load_project` warns on
    hash mismatch; does not block loading (user can re-point).

## Backward Compatibility

- `proj.schema.Material`, `MaterialAssignment`, `Provenance`, `RunResult`
  are **unchanged**. No `SCHEMA_VERSION` bump.
- The Stage A CLI (`raysim run`) continues to work identically with CSV/YAML
  materials + JSON assignments. B2 adds a new code path (library + rules +
  project file) that produces the same `Material[]` and `MaterialAssignment[]`
  the engine already consumes.
- The `raysim.geom.adapter.build_scene_from_step()` signature is unchanged;
  it continues to accept `materials` and `assignments` as explicit arguments.
  The caller (future B3 UI or a project-file-aware CLI extension) resolves
  materials via `raysim.mat` before calling the adapter.
