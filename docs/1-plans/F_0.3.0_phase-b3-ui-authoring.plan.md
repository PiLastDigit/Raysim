# Phase B3 ‚Äî UI + Authoring Implementation Plan

**Version target**: `0.3.0` (minor bump from `0.2.0` ‚Äî adds the `raysim.ui`
package and migrates the STEP loader to XCAF).
**Phase reference**: `MVP_PLAN.md ¬ß6 Phase B3`, `MVP_STEPS.md ¬ßB3.0‚Äì¬ßB3.7`.

## Overview

Phase B3 builds the PySide6 desktop application on top of the headless
engine. It starts with the XCAF migration (B3.0) that unifies the STEP
loader on `STEPCAFControl_Reader`, then builds the main window shell, OCCT
3D viewer, material assignment UI, detector placement, scenario panel, run
dispatcher, and result overlays. This is the first phase that produces a
user-visible product ‚Äî the "internal interactive checkpoint" per
`MVP_PLAN.md ¬ß6`.

## Problem Statement

RaySim's engine (Stage A) and geometry/material layers (B1/B2) exist but are
only accessible via the headless CLI. There is no way to visualize geometry,
interactively assign materials, place detectors, or inspect results. The
`raysim.ui` module is a placeholder. Additionally, the STEP loader still
uses the plain `STEPControl_Reader`, forcing a two-reader workaround for
STEP tag ingestion (documented in B1.1/B2.2 implementation notes). The UI
needs part names and colors that only the XCAF reader provides.

## Solution Architecture

Two layers: B3.0 (XCAF migration) modifies `raysim.geom.step_loader` and
simplifies `raysim.mat.step_tags`. B3.1‚ÄìB3.7 populate `raysim.ui/` with a
PySide6 application. The UI consumes the same APIs the CLI uses ‚Äî no engine
changes.

```
raysim.ui.app          ‚Üê QApplication bootstrap, main window
raysim.ui.viewer       ‚Üê OCCT AIS 3D viewer (pythonocc qtViewer3d)
raysim.ui.panels/
  tree_panel           ‚Üê assembly tree (from AssemblyNode)
  material_panel       ‚Üê library browser + assignment dropdowns
  detector_panel       ‚Üê detector list, placement controls
  scenario_panel       ‚Üê .dos file picker + DDC preview (matplotlib)
  run_panel            ‚Üê run button, progress, cancel
  result_panel         ‚Üê per-detector TID table, diagnostics
raysim.ui.overlays/
  ray_overlay          ‚Üê 3D ray-view colored by mm-Al
  projection_overlay   ‚Üê 6-face unfolded box projection
  mollweide_overlay    ‚Üê matplotlib Mollweide angular shielding
raysim.ui.workers/
  run_worker           ‚Üê QThread wrapping Stage A engine dispatch
```

All UI modules import PySide6 at module level (they only run when the `[ui]`
extra is installed). The `raysim gui` CLI subcommand is the entry point,
guarded by an import check.

## Implementation Details

### 0. XCAF Migration (`raysim.geom.step_loader` + `raysim.mat.step_tags`)
**Files**: `src/raysim/geom/step_loader.py` (modify), `src/raysim/mat/step_tags.py` (modify)

**step_loader.py changes:**
- Replace `STEPControl_Reader` with `STEPCAFControl_Reader` as the primary
  reader. Bootstrap a `TDocStd_Document`, transfer the STEP file, then walk
  the XCAF label tree via `XCAFDoc_DocumentTool.ShapeTool`.
- `LeafSolid` gains three optional fields:
  - `name: str | None` ‚Äî from the XCAF label's `TDataStd_Name` attribute
  - `color_rgb: tuple[float, float, float] | None` ‚Äî from `XCAFDoc_ColorTool`
  - `material_hint: str | None` ‚Äî from `XCAFDoc_MaterialTool`
- `AssemblyNode` gains `name: str | None` from the XCAF label.
- `_build_tree` refactored to walk XCAF labels instead of `TopoDS_Iterator`.
  The DFS walk order and `solid_NNNN` ID assignment remain identical for
  backward compatibility. `path_key` construction unchanged.
- `iter_leaves` unchanged (operates on `AssemblyNode`, not OCCT objects).

**step_tags.py changes:**
- `extract_step_tags(step_path, leaves)` simplified: instead of performing a
  second `STEPCAFControl_Reader` read + two-gate verification, it reads
  `name`, `color_rgb`, and `material_hint` directly from the already-loaded
  `LeafSolid` records. The function becomes a pure mapping from `LeafSolid`
  fields to `StepMaterialTag` instances ‚Äî no OCC imports needed.
- The `_get_material_name`, `_get_color`, `_get_bbox` helpers and the
  verification gate code are removed.
- `extract_step_tags` signature changes to:
  `extract_step_tags(leaves: Sequence[LeafSolid]) -> list[StepMaterialTag]`
  (drops `step_path` parameter ‚Äî no longer reads the file).
- `match_tags_to_library` unchanged.

**Callers to update:**
- `raysim.geom.pipeline.build_assembly_from_step` ‚Äî no change needed, it
  already calls `load_step` and passes leaves downstream.
- Any code calling `extract_step_tags` must drop the `step_path` argument.

**Backward compatibility:** `LeafSolid` and `AssemblyNode` gain optional
fields with `None` defaults. All existing code that destructures these
dataclasses continues to work. The `solid_id` and `path_key` values must
remain identical for the same STEP file ‚Äî project assignments are keyed
by `solid_id`, so an order change would corrupt existing projects.

**DFS order regression test:** before the migration, generate golden fixture
data by running the current B1 `load_step` on every benchmark STEP and
serializing per-leaf `(solid_id, path_key, bbox_min_mm, bbox_max_mm)` tuples
to a checked-in YAML file (`tests/fixtures/step_leaf_golden.yaml`). The B3.0
regression test loads the same STEPs with the new XCAF-based `load_step` and
asserts per-leaf tuples match the golden data exactly. This test is added to
`test_step_loader.py`. If any fixture diverges, the migration must reconcile
the walk order before proceeding.

### 2. OCCT AIS Viewer (`raysim.ui.viewer`)
**File**: `src/raysim/ui/viewer.py` (new)

Wraps pythonocc-core 7.8+ `OCC.Display.qtDisplay.qtViewer3d` widget.

**Responsibilities:**
- `ViewerWidget(QWidget)`: embeds qtViewer3d, exposes `display_shape`,
  `clear`, `fit_all`, `set_view_axis` (front/back/top/bottom/left/right/iso).
- Camera controls: orbit (left-drag), pan (middle-drag), zoom (scroll),
  zoom-to-fit (F key or button).
- Face/edge/vertex picking via AIS selection. Emits
  `shape_selected(solid_id: str, shape_type: str)` signal when user clicks.
- Highlight API: `highlight_solid(solid_id)`,
  `set_solid_color(solid_id, rgb)`, `set_solid_transparency(solid_id, alpha)`.
- Selection state synced bidirectionally with the tree panel.

### 3. Assembly Tree Panel (`raysim.ui.panels.tree_panel`)
**File**: `src/raysim/ui/panels/tree_panel.py` (new)

`QTreeWidget` displaying the `AssemblyNode` hierarchy from `load_step`.
Each node shows: `name` (from XCAF, or `solid_id` fallback), material
assignment status (icon: green check / yellow warning / red X), color swatch
from XCAF color.

**Interactions:**
- Click ‚Üí selects in viewer (bidirectional sync via signals).
- Right-click context menu: "Assign material...", "Propagate to subtree",
  "Show in viewer", "Properties".
- Drag-and-drop material from the material panel onto a tree node.

### 4. Material Panel (`raysim.ui.panels.material_panel`)
**File**: `src/raysim/ui/panels/material_panel.py` (new)

Shows the seeded material library + project-local additions. Wired to B2's
`MaterialLibrary`, `AssignmentReview`, and `build_review()`.

**Layout:**
- Top: library table (`QTableWidget`) with columns: group_id, display_name,
  density_g_cm3, z_eff, assignment count.
- Middle: assignment status bar (auto-matched / ambiguous / unassigned counts
  from `AssignmentReview`).
- Bottom: action buttons ‚Äî "Accept all suggestions", "Review ambiguous",
  "Clear all assignments".

**Assignment workflow:**
- On STEP open: runs `apply_rules` + `extract_step_tags` (via B2 review API)
  ‚Üí populates the tree with auto-assignments.
- User reviews ambiguous/unassigned via the tree panel or material panel.
- Dropdown per tree node (in tree panel) selects from the library.
- "Accept all" resolves everything that has a single-candidate match.
- Run gating status updated in real time via `check_run_readiness`.

### 5. Detector Panel (`raysim.ui.panels.detector_panel`)
**File**: `src/raysim/ui/panels/detector_panel.py` (new)

`QListWidget` of placed detectors + placement controls.

**Placement modes:**
- Click-pick on a CAD face ‚Üí places detector at face centroid.
- Snap modes: face centroid (default), vertex, edge midpoint, free position.
  Selected via small toolbar or modifier keys (Shift = vertex, Ctrl = edge
  midpoint, no modifier = centroid).
- Normal-offset `QDoubleSpinBox` (0‚Äì100 mm) pushes detector along face normal.
- **Box template deferred.** The current aggregator only supports point
  detectors. Box-to-subdetector sampling (MVP_PLAN ¬ß4.9) requires engine
  changes that are out of B3 scope. The detector panel shows point-only
  placement in B3; box template UI + engine support land together in a
  future phase.

**List interactions:**
- Select ‚Üí highlights in viewer with a detector glyph (sphere).
- Right-click: rename, delete, inspect (shows position).
- Detectors persisted in `ProjectFile.detectors` via `Detector` model.

### 6. Scenario Panel (`raysim.ui.panels.scenario_panel`)

### 7. Run Panel + Dispatcher (`raysim.ui.panels.run_panel` + `raysim.ui.workers.run_worker`)
**Files**: `src/raysim/ui/panels/run_panel.py` (new), `src/raysim/ui/workers/run_worker.py` (new)

**run_panel.py:**
- "Run" button (enabled only when `check_run_readiness().ready` is True).
- `QProgressBar` driven by per-detector completion (0‚Äìn_detectors).
- "Cancel" button sends a cancellation signal to the worker.
- Status label: "Ready" / "Running (detector 3/10)" / "Complete" / "Cancelled".
- On completion: populates result panel, writes `run.json`, enables overlays.

**run_worker.py:**
- `RunContext`: frozen dataclass carrying all inputs needed for a run **and**
  the pre-computed provenance hashes. Built by the app controller from the
  current project state before dispatching: `scene: BuiltScene`,
  `spline: DoseSpline`, `detectors: list[Detector]`, `nside: int`,
  `emit_pixel_map: bool`, `output_path: Path`, plus
  `geometry_hash: str`, `materials_hash: str`, `assignments_hash: str`,
  `detectors_hash: str`, `dose_curve_hash: str`. This ensures the GUI
  builds bit-identical provenance to the CLI (ARCHI ¬ß14) without
  reconstructing hashes inside the thread.
- `RunWorker(QThread)`: takes a `RunContext`, runs the per-detector
  aggregation loop.
- Emits `detector_done(index: int, result: DetectorResult)` signal per detector.
- Emits `run_complete(run_result: RunResult)` on finish.
- Emits `run_error(message: str)` on failure.
- Supports cancellation via `QThread.requestInterruption()` checked between
  detector iterations.

**Provenance:** `RunWorker` builds the `Provenance` block from `RunContext`'s
pre-computed hashes plus `__version__`, `_library_versions()`, `nside`,
`epsilon_mm`, and `bbox_diag_mm` from the scene. The `run.json` output is
identical whether produced via CLI or GUI.

### 8. Result Panel (`raysim.ui.panels.result_panel`)
**File**: `src/raysim/ui/panels/result_panel.py` (new)

Shows per-detector results after a run completes.

**Layout:**
- `QTableWidget` with columns: detector name, dose_total_krad, mm-Al mean,
  angular spread, n_stack_leak, n_overlap_suspicious.
- Select a detector row ‚Üí updates overlays for that detector.
- Per-species breakdown expandable per row.
- Health diagnostic summary at the bottom (total stack leaks, max-hit rays).

### 9. Result Overlays (`raysim.ui.overlays/`)
**Files**: `src/raysim/ui/overlays/ray_overlay.py` (new),
`src/raysim/ui/overlays/projection_overlay.py` (new),
`src/raysim/ui/overlays/mollweide_overlay.py` (new)

Three overlays, toggleable per detector via checkboxes in the result panel.

**ray_overlay.py:**
- Renders 3D lines from the detector position along each HEALPix direction,
  colored by accumulated mm-Al-equivalent (blue = thin, red = thick).
- Uses OCCT AIS `AIS_Line` or `AIS_Shape` objects added to the viewer.
- Clicking a ray line shows that pixel's ‚àëœÅL, mm-Al, and dose in a tooltip
  (data already available from `healpix_mm_al_per_pixel`). Per-ray hit-list
  detail requires a debug-trace API not yet in the engine ‚Äî deferred.
- Length of rendered ray = first-miss distance (or bbox_diag for LOS rays).

**projection_overlay.py:**
- 6-face equivalent-thickness projection rendered as a 2D unfolded box in
  a separate `QDialog` or docked panel.
- Each face shows the mean mm-Al for HEALPix pixels in that hemisphere
  quadrant, as a color-mapped rectangle.
- Matplotlib `FigureCanvas` with 6 subplots (top/bottom/front/back/left/right).

**mollweide_overlay.py:**
- Matplotlib-only Mollweide projection of the per-pixel mm-Al array.
- Uses `matplotlib.pyplot.subplot(projection='mollweide')` with `pcolormesh`
  over the HEALPix pixel grid (RING ordering ‚Üí theta/phi conversion).
- Colorbar: mm-Al-equivalent. Graticule lines at 30-degree intervals.
- Renders in a docked panel or `QDialog`.
- Clicking a pixel shows that pixel's mm-Al value and dose from the spline
  (computed on the fly from the per-pixel array). Per-ray hit list deferred.
- **Future optimization path:** matplotlib renders ~200ms per detector at
  Nside=64. If post-MVP users need faster rendering at high Nside (e.g.
  Nside=256, ~1.5s per detector), a `healpy.mollview` fast path can be
  added behind a runtime dispatch (`healpy` available ‚Üí use it, else ‚Üí
  matplotlib fallback). This is a single-function optimization, not an
  architectural change ‚Äî the overlay API and data flow are identical. The
  matplotlib path is the MVP baseline because it works on all platforms
  including Windows where healpy may not install.

### 11. Application State Controller (`raysim.ui.state`)
**File**: `src/raysim/ui/state.py` (new)

Central owner of the mutable application state. All panels read from and
write to this controller ‚Äî they do not hold independent copies.

**Responsibilities:**
- `AppState(QObject)`: holds the current `ProjectFile` (or None for unsaved),
  the loaded `AssemblyNode` tree, `BuiltScene`, `MaterialLibrary`,
  `AssignmentReview`, `DoseSpline`, `list[DetectorResult]`, and run output.
- **Project lifecycle:** `new_project()`, `open_project(path)`,
  `save_project()`, `save_project_as(path)`. Manages the `ProjectFile`
  instance, resolves relative paths against the project directory.
- **Dirty tracking:** `is_dirty: bool` property. Set when assignments,
  detectors, scenario, or naming-rule overrides change. Cleared on save.
  "Close" / "Exit" prompts to save when dirty.
- **Geometry lifecycle:** `open_step(path)` calls `load_step` ‚Üí pipeline ‚Üí
  `build_scene_from_step`, populates the tree and scene, runs auto-assignment
  via B2 review API. `reload_step()` re-opens the current STEP (e.g. after
  external CAD edit).
- **Signals:** `project_changed`, `scene_loaded`, `assignments_changed`,
  `detectors_changed`, `scenario_changed`, `run_started`, `run_complete`.
  Panels connect to these for reactive updates.
- **Run context builder:** `build_run_context() -> RunContext` computes all
  provenance hashes from the current state and returns the frozen context
  for `RunWorker`.

## Technical Considerations

- **Pattern Usage**: follows `frozen=True` + `extra="forbid"` for any new
  Pydantic models. UI state classes use regular `QObject`-derived classes
  (not frozen ‚Äî UI state is mutable by nature).

- **Determinism Impact**: none. The UI produces the same `RunResult` via the
  same engine path. `run.json` output is identical whether from CLI or GUI.
  The `Provenance` block is built identically by `run_worker`.

- **Numerical Precision**: no impact. B3 does not touch the ray engine, dose
  math, or float accumulation paths.

- **Material Physics Scope**: unchanged. The UI exposes the same
  `density_g_cm3`-only material model.

- **Coincident-Face / Tied-Batch**: no impact. The viewer renders geometry
  but does not modify the BVH or traversal.

- **Backend Optionality**: `raysim.ui` requires PySide6 (`[ui]` extra) and
  pythonocc-core (conda). The `raysim gui` command fails gracefully with a
  clear error message if PySide6 is missing. B3.0 (XCAF migration) is
  OCCT-dependent ‚Äî same `importorskip` pattern as B1.

- **Schema Versioning**: no `SCHEMA_VERSION` bump (run-output schema
  unchanged). No `PROJECT_SCHEMA_VERSION` bump (`ProjectFile` fields
  unchanged ‚Äî the UI reads/writes the same schema B2 defined).

- **Edge Cases**:
  - Opening a STEP with 0 solids ‚Üí empty tree, run gating blocks.
  - Detector placed outside all geometry ‚Üí enclosing-solids probe returns
    empty, all rays start with empty stack (valid ‚Äî measures external
    shielding).
  - Cancel mid-run ‚Üí partial results discarded, no `run.json` written.
  - `.dos` file with parse error ‚Üí scenario panel shows error message, run
    gating blocks.
  - Large STEP (50M+ triangles) ‚Üí tessellation may be slow; progress
    indication during open is desirable but not gated.
  - Project file referencing missing STEP ‚Üí geometry hash warning on load,
    viewer shows nothing, user can re-point.

## Performance & Cost Impact

- **STEP open time**: XCAF reader is ~10‚Äì20% slower than plain reader on
  large files (more metadata to parse). Acceptable for an interactive app
  where the user waits for "Open" to complete.
- **Overlay rendering**: Mollweide at Nside=64 ~200ms per detector
  (matplotlib). 3D ray-view at 49K rays ~500ms (AIS line creation). Both
  within the 1s toggle budget. Rendering is per-detector on user selection,
  not batch ‚Äî even with 30 detectors, the user sees one overlay at a time.
  At Nside=256 (~1.5s/detector), matplotlib is the bottleneck; a future
  `healpy.mollview` fast path (~300ms) can be added post-MVP if needed.
- **Run dispatch**: identical performance to CLI (same engine path). QThread
  adds negligible overhead.

## Backward Compatibility

- `LeafSolid` and `AssemblyNode` gain optional fields with `None` defaults.
  All existing code continues to work without changes.
- `extract_step_tags` loses the `step_path` parameter. All callers must
  update (only `raysim.mat.step_tags` tests and any downstream code calling
  the function directly).
- `raysim run` CLI continues to work identically. The GUI is additive.
- `ProjectFile` schema unchanged ‚Äî no version bump needed.
