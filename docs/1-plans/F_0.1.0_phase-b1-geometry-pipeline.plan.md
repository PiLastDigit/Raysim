# Phase B1 — Geometry Pipeline Implementation Plan

**Version target**: `0.1.0` (minor bump from `0.0.2` — adds the `raysim.geom`
package, the first Stage B surface).
**Phase reference**: `MVP_PLAN.md §6 Phase B1`, `MVP_STEPS.md §B1.1–§B1.6`.

## Overview

Phase B1 turns a single STEP file into the same in-memory `BuiltScene`
that Phase A's `load_scene_from_directory` produces, so the existing
Stage A engine plugs in unchanged. The pipeline is: STEP read → leaf-solid
walk → per-solid tessellation → healing + shell-orientation normalization
→ per-shell watertightness validation → solid-pair overlap/interference
diagnostic → STL export → handoff to `raysim.ray.scene.load_scene_from_directory`.

The whole layer sits behind `pythonocc-core` (conda-forge `raysim-occt`
micromamba env). Default `uv run pytest` skips B1 tests via
`pytest.importorskip('OCC')`; running B1 tests is `micromamba run -n
raysim-occt pytest -m needs_occt`.

## Problem Statement

Stage A consumes geometry as a directory of STLs. Stage B's user-facing
flow opens a STEP file. Until B1 lands, there is no path from STEP to
the engine. This phase adds the geometry kernel layer — STEP ingest,
tessellation, healing, validation, diagnostics — *without* modifying the
Stage A engine. Per `docs/decisions/phase-0.md`:

- `pythonocc-core` 7.9.0 ships in conda-forge only.
- `BRepMesh_IncrementalMesh` and `BRepMesh_ModelHealer` are exposed and
  importable — verified by `tests/unit/test_occt_smoke.py`.
- `XCAFDoc_DocumentTool` bootstrap throws `Standard_NullObject` —
  documented xfail in the same smoke file. **Consequence**: B1.1 uses
  the non-CAF `STEPControl_Reader` path. Per-part names, colors, and
  layers from XCAF land with B2.2 once the bootstrap is unblocked. B1
  uses synthetic stable solid IDs (`solid_0001`, `solid_0002`, …) keyed
  by walk order in the STEP compound.

## Solution Architecture

```
STEP file
    │ STEPControl_Reader (B1.1)
    ▼
TopoDS_Compound — walked deterministically into a list of TopoDS_Solid leaves
    │ BRepMesh_IncrementalMesh per leaf (B1.2)
    ▼
TessellatedSolid[]  (per-shell triangle arrays, float64)
    │ BRepMesh_ModelHealer + shell-classification + probe-ray normalization (B1.3)
    ▼
HealedAssembly
    │ Per-shell watertightness: edge-pairing check (B1.4)
    │ Solid-pair overlap diagnostic: AABB pre-filter + planar-face coplanar
    │     test + BRepAlgoAPI_Common (face overlap) + BOPAlgo_BOP Common
    │     (volume) (B1.5)
    ▼
ValidatedAssembly  (+ OverlapReport, WatertightnessReport)
    │ Per-leaf STL export → tempdir; assignments JSON → tempdir (B1.6)
    ▼
load_scene_from_directory(tempdir, materials, assignments)
    ▼
BuiltScene  (Stage A engine consumes unchanged)
```

The pipeline is orchestrated by `raysim.geom.pipeline.build_assembly_from_step()`
returning a `ValidatedAssembly`; `raysim.geom.adapter.build_scene_from_step()`
threads this into the Stage A loader.

## Implementation Details

### 1. `raysim.geom.step_loader` (B1.1)

**File**: `src/raysim/geom/step_loader.py` (new)

**What it does**
- Wraps `OCC.Core.STEPControl.STEPControl_Reader`. Reads the file, returns
  the root `TopoDS_Shape`.
- Walks the root compound recursively. Each level emits an
  `AssemblyNode` (children list + the level's `TopoDS_Shape`); leaves are
  `LeafSolid` records carrying the `TopoDS_Solid`, a stable
  `solid_id` (`solid_NNNN`, zero-padded width 4 by default), and the
  walk-path-key (e.g. `"0/2/1"`).
- Computes per-leaf bbox in mm directly from `TopoDS_Solid` vertices via
  `BRepBndLib.Add` — used for the B1.5 AABB pre-filter and for sanity
  checks.

**Public surface**
- `class AssemblyNode` (frozen dataclass): `path_key: str`, `children:
  tuple[AssemblyNode, ...]`, `leaf: LeafSolid | None`.
- `class LeafSolid` (frozen dataclass): `solid_id: str`, `path_key: str`,
  `shape: object` (`TopoDS_Solid`), `bbox_min_mm: tuple[float, float,
  float]`, `bbox_max_mm: tuple[float, float, float]`.
- `def load_step(path: str | Path) -> AssemblyNode: ...`
- `def iter_leaves(node: AssemblyNode) -> Iterator[LeafSolid]: ...` —
  deterministic depth-first, walk-order stable.

**Deferred to B2.2**: per-part names, colors, layers (need `STEPCAFControl_Reader`
+ `XCAFDoc_*Tool`, blocked by the phase-0 bootstrap regression). The xfail
test in `tests/unit/test_occt_smoke.py::test_xcafdoc_material_tool_bootstrap`
is the trigger that unblocks this.

### 2. `raysim.geom.tessellation` (B1.2)

**File**: `src/raysim/geom/tessellation.py` (new)

**What it does**
- Runs `BRepMesh_IncrementalMesh(shape, linear_deflection, theAngDeflection,
  isRelative=False, isInParallel=True)` on each `TopoDS_Solid`.
  `BRepMesh` *attaches* the triangulation to the shape in place; the
  healer reads it back from the same shape, so the `TopoDS_Solid` handle
  must stay alive through B1.3.
- Defaults: `linear_deflection_mm = 0.1`, `angular_deflection_rad = 0.5`
  per `MVP_STEPS §B1.2`.
- Walks `TopExp_Explorer(solid, TopAbs_FACE)` to extract per-face
  `BRep_Tool.Triangulation`; groups by source `TopoDS_Shell` (parent of
  the face) so the watertightness validator can operate per shell.
- Output: `TessellatedSolid` carrying per-shell `TessellatedShell`
  records — each with its own `vertices` (`(n_v, 3)` float64), `faces`
  (`(n_t, 3)` int64), and `triangle_normals` (`(n_t, 3)` float64
  unit-length). Per-shell separation is the source-of-truth for
  the watertightness validator (which operates per shell). The
  per-solid flat triangle index is *derived* from the shell ordering
  via the `flatten_index` helper below — there is no flat
  `shell_index_per_triangle` array on `TessellatedSolid`.
- The output also carries the source `TopoDS_Solid` handle on
  `TessellatedSolid.shape` so B1.3's healer can operate on the same
  shape and re-extract the triangulation post-heal without reloading
  the STEP.

**Triangle normals** computed in float64 from the source vertex positions,
unit-normalized. The face-orientation flag (`TopAbs_FORWARD` /
`TopAbs_REVERSED`) is honored at this layer (flips winding accordingly)
but the result is **not yet trusted** — B1.3's probe-ray validator
verifies it after healing.

**Public surface**
- `class TessellatedShell`: per-shell `vertices`, `faces`,
  `triangle_normals`, `shell_index: int`.
- `class TessellatedSolid`: `solid_id`, `shape: object  # TopoDS_Solid`,
  `shells: tuple[TessellatedShell, ...]`, `bbox_min_mm`, `bbox_max_mm`.
  The `shape` field is the OCCT handle that B1.3 needs to drive the
  healer; B1.3's `HealedSolid` drops it once healing is complete.
- `def tessellate(leaf: LeafSolid, *, linear_mm: float = 0.1,
  angular_rad: float = 0.5) -> TessellatedSolid: ...`

**Per-solid global triangle index convention** (the contract every
downstream module obeys):

A solid's *flat* triangle index — used as `prim_a` / `prim_b` in
`TiedTrianglePair` and as the input domain of B1.6's
`triangle_index_map` — is the concatenation of per-shell `faces`
arrays in **shell-index order**:

```
flat_index_of(shell_index=k, prim_in_shell=j) =
    sum(len(shells[i].faces) for i in 0..k-1) + j
```

`shell_index` is itself stable: it's the order in which `TopExp_Explorer(solid,
TopAbs_SHELL)` walks shells of the source `TopoDS_Solid`. Healed solids
preserve shell ordering; flipped shells (B1.3) keep their `shell_index`.

This convention applies to `TessellatedSolid`, `HealedSolid`, the STL
exporter, and the `TiedTrianglePair` indices. B1.6's
`ExportedSolid.triangle_index_map[flat_idx] = export_idx` translates
the flat index into the post-export Embree `prim_id` (which is the
post-lex-sort STL triangle order).

A small helper `def flatten_index(solid: TessellatedSolid | HealedSolid,
shell_index: int, prim_in_shell: int) -> int` is exposed on the public
surface so callers don't reimplement the prefix-sum.

### 3. `raysim.geom.healing` (B1.3)

**File**: `src/raysim/geom/healing.py` (new)

**What it does (two passes)**

**Pass 1 — `BRepMesh_ModelHealer`**
- Operates on the `TopoDS_Solid` handle that B1.2 carries forward on
  `TessellatedSolid.shape` — the healer modifies the triangulation
  attached to the shape in place. Apply
  `OCC.Core.BRepMesh.BRepMesh_ModelHealer.PerformAll` (or the
  per-solid Healer per the OCCT 7.9 binding).
- After the healer runs, **re-extract** the triangulation from the same
  shape (same per-face / per-shell walk B1.2 does) — the freshly-healed
  triangles replace the pre-heal arrays.
- Idempotency invariant: running the healer twice in a row produces no
  further changes (verified by a unit test on `aluminum_box.step` and
  `concentric_shell.step`).
- Log the diff: shells touched, micro-cracks closed, edges snapped,
  degenerate triangles removed.

**Pass 2 — Shell-orientation normalization**

The convention RaySim enforces (per `MVP_STEPS §B1.3` and `ARCHI §11`):
**every triangle's normal points out of solid material** — into vacuum
for outer shells, into the cavity void for cavity shells. A
perfectly-watertight inward-oriented shell silently inverts the
A.4 entry/exit classification → wrong ∑ρL.

Algorithm per `TessellatedSolid` — **single-probe approach** (avoids
chicken-and-egg of using normals to find an "interior" point on a
shell whose normals may themselves be wrong):

1. **Sub-shell classification.** For each closed shell of the solid,
   compute the **vertex centroid** (average of all vertices — geometric,
   normal-independent). Test point-in-volume containment against every
   other shell of the same solid using a trimesh ray-cast (odd-crossing
   rule based on triangle vertex positions, *not* normals). The shell
   containing all other shells' vertex centroids is the *outer*; the
   rest are *cavity* shells.
   - **Single-shell solid**: if `len(shells) == 1`, the single shell is
     the outer; no cavity classification needed.
   - **Multi-cavity solids** (one outer, N cavities) supported.

2. **Single through-the-solid probe.** Pick a probe ray from a point
   provably outside the solid (outside the bbox along +X) toward the
   solid's bbox centroid. Cast it through the solid's combined
   triangle set (all shells) and record the ordered hits with their
   per-hit `(shell_index, dot_product = dot(ray_dir, triangle_normal))`.
   - If a hit's `|dot| < 1e-9` (tangent / grazing), retry with a
     fallback direction (+Y, then +Z); log if all three fail.

3. **Expected pattern from the probe** (for a correctly-oriented
   solid under RaySim's "out of solid material" convention):
   - The hit sequence alternates `entry, exit, entry, exit, …` with
     stack count returning to zero at the end (ray exits the solid).
   - Outer-shell hits going *into* the solid have `dot < 0` (entry);
     outer-shell hits going *out* have `dot > 0` (exit).
   - Cavity-shell hits going *into* the cavity (from solid material
     into void) have `dot > 0` (exit, because the ray is leaving solid
     material); cavity-shell hits going *out* of the cavity have
     `dot < 0` (entry into solid material).

4. **Per-shell flip decision.** Examine the hit sequence:
   - **Outer shell**: the first hit on the outer shell along the probe
     must be `entry` (`dot < 0`). If it's `exit`, the outer shell is
     reversed → flip every outer-shell triangle's vertex order (and
     recompute normals). Re-cast and re-verify.
   - **Cavity shells**: after fixing the outer (if needed), the
     post-outer-flip hit pattern is examined. For each cavity shell,
     the first hit on that shell must be `exit` (`dot > 0`, ray going
     from solid into cavity). If it's `entry`, the cavity shell is
     reversed → flip every triangle of that cavity shell.
   - **Critical**: cavity-shell normals correctly *point into the
     cavity void* — they should look "inverted" relative to a
     standalone closed volume, but the dot-product against an outward-
     traveling probe is what determines correctness, not visual
     intuition. Do **not** flip cavity shells based on normal
     direction alone.

5. **Re-verification.** After all per-shell flips, re-cast the same
   probe; the entire sequence must follow `entry, exit, entry, exit,
   …` ending with stack count zero. Failure here is a `B1.3 healing
   error` logged with the offending solid/shell.

6. **Expected pattern checks** for verification fixtures:
   - Solid no-cavities (single shell): `entry, exit` from external
     probe.
   - Solid one-cavity (two shells): `entry, exit, entry, exit` —
     material-void-material-void as the ray crosses the cavity.

**Public surface**
- `class ShellRole(StrEnum)`: `OUTER`, `CAVITY`.
- `class HealedShell`: extends `TessellatedShell` with `role: ShellRole`,
  `was_flipped: bool`.
- `class HealedSolid`: per-solid post-healing record + per-shell role.
- `def heal_assembly(solids: Sequence[TessellatedSolid]) ->
  tuple[HealedSolid, ...]`.

### 4. `raysim.geom.watertightness` (B1.4)

**File**: `src/raysim/geom/watertightness.py` (new)

**What it does**
- Per healed shell: build the edge-pair map. Edge = sorted pair of
  vertex indices. Every non-degenerate edge must appear in exactly
  two triangles, and the two triangles' vertex orders for that edge
  must be opposite (one `(v_a, v_b)`, the other `(v_b, v_a)`).
- Degenerate edges (zero length within tolerance) are dropped before
  the count.
- Failure modes: `unpaired_edge` (count != 2), `same_orientation`
  (two triangles share an edge with the same vertex order, i.e. a
  Möbius-strip topology), `degenerate_triangle` (zero area).

**Public surface**
- `class ShellWatertightness`: `shell_index`, `pass_: bool`,
  `unpaired_edges: tuple[tuple[int, int], ...]`,
  `same_orientation_edges: tuple[tuple[int, int], ...]`,
  `degenerate_triangles: tuple[int, ...]`.
- `class WatertightnessReport`:
  - `per_solid: Mapping[str, tuple[ShellWatertightness, ...]]`
  - `def failed_shells(self) -> tuple[tuple[str, int], ...]` — list
    of `(solid_id, shell_index)` for every shell where `pass_ is
    False`. Empty tuple ⇒ assembly is wholly watertight. The B1.6
    adapter calls this to decide whether to gate.
  - `def is_watertight(self) -> bool` — convenience.
- `def validate_watertightness(solids: Sequence[HealedSolid]) ->
  WatertightnessReport`.

### 5. `raysim.geom.overlap` (B1.5)

**File**: `src/raysim/geom/overlap.py` (new)

**What it does (face-level coplanar classification + triangle-level vertex-match tie pairing + four-way solid-pair status)**

The tied-group payload that flows into the engine is **size-2 pairs of
vertex-matching triangles**, not "every triangle on the contact face."
The Stage A tracer's tied-batch contract is "all members are at the
same 3D point within float-noise; the ray crosses all of them
simultaneously, processed as zero-length push/pop." Putting an entire
contact face's triangles into one group would trigger push/pop for
triangles the ray isn't crossing at this `t` ⇒ stack mismatches ⇒
wrong ∑ρL. So tied groups stay triangle-level (two coincident
triangles per group, like Stage A's existing detector), while the
*classification* of the pair (`contact_only` / `interference_*`) is
done at the OCCT face / volume level.

**Scope decision for B1**: this phase handles only the
**topology-shared** contact case — two solids whose contact faces
share the underlying `TopoDS_Face` in the STEP, so `BRepMesh` produces
**identical triangulations** on both sides (the realistic case for
clean STEP assemblies; what `concentric_shell.step` gives us). The
detector pairs vertex-matching triangles from opposite sides into
size-2 tied groups, exactly matching Stage A's vertex-set hash detector.

The **mismatched-tessellation** case (two solids that touch
geometrically but were tessellated independently with non-aligning
triangle vertices) is detected and reported as a separate
`MismatchedContactRegion` warning, *not* converted to tied groups. The
adapter's `accept_warnings` gate covers it. **Mesh-Boolean
re-triangulation of mismatched contact regions is deferred to a
follow-up phase** (B1.5b or B5.x). The plan ships clear detection +
explicit user signal + override, not a silent partial fix.

**Detection passes**

- **Solid-solid AABB pre-filter.** O(N²) pairwise; cheap. Only
  AABB-overlapping pairs proceed to the expensive checks below.
- **Face-level coplanar contact detection** (per AABB-overlapping pair):
  1. Walk `TopExp_Explorer(solid, TopAbs_FACE)` on both solids. **Skip
     non-planar faces.** Coplanarity is only well-defined for
     `Geom_Plane` surfaces (or `Geom_RectangularTrimmedSurface` /
     `Geom_OffsetSurface` whose underlying surface is a plane).
     Curved faces (cylinders, cones, spheres, splines) are
     out of scope for B1's coplanar-contact detection — most realistic
     contact surfaces are planar (mounted boards, machined mating
     faces). A non-planar contact (e.g., shaft-in-bore) that produces
     a non-zero intersection volume falls through to the volume-based
     `interference_*` classification, which is the correct outcome.
  2. For each pair of planar faces `(F_a, F_b)`, test coplanarity:
     surface normals **parallel or anti-parallel** (anti-parallel is the
     typical case because outward normals on a shared material boundary
     point in opposite directions) within `COPLANAR_NORMAL_TOL_RAD =
     1e-6` *and* common plane (signed distance between any vertex of
     `F_b` and the plane of `F_a` below `COPLANAR_PLANE_TOL_REL ×
     bbox_diag = 1e-6 × bbox_diag`).
  3. For each coplanar pair, compute the **shared planar region** with
     `BRepAlgoAPI_Common(F_a, F_b)`. Discard empty (disjoint within
     the plane). If `BRepAlgoAPI_Common` reports `IsDone() == False`,
     record the pair on `OverlapReport.boolean_failures` and proceed
     (per the edge-cases section).
  4. Project both faces' triangulated triangle vertices onto the
     shared plane (3D → 2D via an orthonormal basis on the plane).
- **Triangle-level vertex-match pairing** (per shared planar region):
  Hash each triangle of either side by **the unordered set of its
  three projected 2D vertex positions** (rounded to
  `VERTEX_MATCH_TOL_REL × bbox_diag = 1e-6 × bbox_diag`, then sorted
  lex on `(x_rounded, y_rounded)`). Two triangles with the same hash
  on opposite sides form a `TiedTrianglePair` (size-2 tied group).
  Vertex-set hashing means winding orientation is **not** part of the
  match — opposite-winding triangles (the typical case for two solids'
  outward faces on a shared boundary) hash equal, which is correct.
  As a sanity guard, the matched pair's source triangle normals must
  satisfy `dot(n_a, n_b) < -1 + COPLANAR_NORMAL_TOL_RAD` (anti-parallel
  on opposite sides of the shared boundary). A pair that hashes equal
  but fails the anti-parallel check is a degenerate case (logged,
  not paired).
- **Coverage check on the shared planar region.** Compute the matched
  area = sum over `TiedTrianglePair`s of the smaller of the two
  triangles' projected 2D area, divided by the OCCT-computed shared
  region's total area. If `matched_area_fraction < 1.0 -
  COVERAGE_EPSILON` (default `COVERAGE_EPSILON = 1e-3`, i.e. less
  than 99.9% of the contact polygon is covered by vertex-matched
  pairs), record a `MismatchedContactRegion` warning. **Any non-trivial
  untied subregion on a coplanar material boundary can produce stack
  errors** — partially-matched contacts (50%, 90%, 99%) all fall in
  the same correctness hole as fully-mismatched ones, so the gate
  triggers on any coverage shortfall, not only on near-zero matches.
  - Tied pairs *that did match* are still emitted; the warning is
    additive — the user is told "this region has good pairs A-N and a
    leftover untied subregion".
  - Without `accept_warnings=True` override, the run is blocked.
    With override, the matched pairs flow through to the engine and
    the leftover subregion is left untied (the user explicitly
    accepts that traversal in those triangles is not contact-aware).
- **Solid-solid intersection volume.** Per AABB-overlapping pair,
  `BOPAlgo_BOP` (`Common`) computes the intersection volume via
  `OCC.Core.GProp.GProp_GProps` + `BRepGProp.VolumeProperties`. Volume
  below `1e-9 × min(V_a, V_b)` ⇒ no penetration (only face-level
  contact).
- **Nested-solid detection.** If `volume(A ∩ B) ≈ min(volume(A),
  volume(B))` within `1e-6` relative tolerance, the smaller solid is
  fully inside the larger.

**Why this approach is correct for B1's scope**:
- Topology-shared STEPs (clean assemblies, the realistic input) get
  exact triangle-level pairing — the engine sees size-2 groups, the
  tracer's tied-batch invariants hold.
- Mismatched-tessellation contacts are *detected*, *reported*, and
  *blocked by default* — the user is never silently given wrong
  ∑ρL. With explicit override the user accepts the bias, knowing it
  comes from coplanar surfaces that don't align triangle-by-triangle.
- The mesh-Boolean fix (re-triangulating both sides to a shared mesh
  in the contact region) is well-scoped for a follow-up phase and
  doesn't block B1's deliverable.

**Four-way solid-pair classification** (mutually exclusive, by volume):

| Status | Detection | Stack effect | Default action |
|---|---|---|---|
| `contact_only` | Face-level coplanar contact + intersection volume ≈ 0 | Vertex-match tie pairs transition A↔B cleanly | Accept silently |
| `accepted_nested` | Volume(A∩B) ≈ Volume(smaller) within tol | Stack carries both A and B inside B's region — physically correct | Accept; surface in report |
| `interference_warning` | 0 < V_int < min(1 mm³, 0.1% of V_smaller) | ρ_A + ρ_B inside overlap — biased but small | Block by default; accept on `accept_warnings=True` |
| `interference_fail` | V_int > the above thresholds | Same bias, large enough to matter | Block; require CAD fix |

`MismatchedContactRegion` is a **separate warning category**, parallel
to (not part of) the four pair-status values; it can fire alongside any
of them when a coplanar contact has nonzero area but no vertex-match
pairs. It is gated by the same `accept_warnings` flag the adapter
uses for `interference_warning`.

**Public surface**

```python
class OverlapStatus(StrEnum):
    CONTACT_ONLY = "contact_only"
    ACCEPTED_NESTED = "accepted_nested"
    INTERFERENCE_WARNING = "interference_warning"
    INTERFERENCE_FAIL = "interference_fail"

@dataclass(frozen=True)
class TiedTrianglePair:
    """One size-2 tied group: a triangle in solid_a coincident (within
    float-noise on a shared plane) with a triangle in solid_b."""
    solid_a: str
    prim_a: int           # per-solid flat triangle index — see §2 convention
    solid_b: str
    prim_b: int           # per-solid flat triangle index — see §2 convention

@dataclass(frozen=True)
class MismatchedContactRegion:
    """Coplanar contact between two solids whose vertex-match coverage
    falls below 1.0 - COVERAGE_EPSILON of the shared region. Any
    shortfall — fully mismatched (0.0), partially matched (e.g. 0.5),
    or near-but-not-quite full (0.99) — is correctness-equivalent on
    the untied subregion, so the warning fires for any value below the
    threshold. The adapter blocks unless accept_warnings=True.
    Matched pairs in the same region are still emitted as
    TiedTrianglePairs on the parent OverlapPair; the warning is
    additive."""
    solid_a: str
    solid_b: str
    face_a_index: int     # TopExp index on solid_a
    face_b_index: int     # TopExp index on solid_b
    shared_area_mm2: float
    matched_area_fraction: float  # 0.0 = fully mismatched; <1.0 = partial

@dataclass(frozen=True)
class OverlapPair:
    solid_a: str
    solid_b: str
    status: OverlapStatus
    intersection_volume_mm3: float
    bias_estimate_g_per_cm2: float
    tied_triangle_pairs: tuple[TiedTrianglePair, ...]
    """All vertex-match tied pairs found between this solid pair.
    Empty for accepted_nested and interference_* with no coplanar
    faces; size-N for contact_only."""

@dataclass(frozen=True)
class BooleanFailure:
    """Recorded when BRepAlgoAPI_Common or BOPAlgo_BOP returns
    IsDone() == False. The pair is treated as undetermined (gated
    under accept_warnings) rather than crashing the pipeline."""
    solid_a: str
    solid_b: str
    operation: str       # "common_face" | "common_volume"
    occt_message: str    # whatever OCCT exposes via DumpErrors / GetReport

@dataclass(frozen=True)
class OverlapReport:
    pairs: tuple[OverlapPair, ...]
    mismatched_contacts: tuple[MismatchedContactRegion, ...]
    boolean_failures: tuple[BooleanFailure, ...]

    def failed(self) -> tuple[OverlapPair, ...]: ...      # interference_fail
    def warnings(self) -> tuple[OverlapPair, ...]: ...    # interference_warning
    def all_tied_triangle_pairs(self) -> tuple[TiedTrianglePair, ...]: ...
        # Flattens across all OverlapPair.tied_triangle_pairs for the adapter.

def diagnose_overlaps(
    solids: Sequence[HealedSolid],
    *,
    # Volume-based classification thresholds — exposed so the user can
    # tune them per project (e.g. tighter on a precision instrument).
    small_volume_threshold_mm3: float = 1.0,
    small_relative_volume: float = 1e-3,
    # Geometry-tolerance group — module-level constants (see below);
    # exposed as kwargs so a user with unusual scale can override.
    coplanar_normal_tol_rad: float = COPLANAR_NORMAL_TOL_RAD,    # 1e-6
    coplanar_plane_tol_relative: float = COPLANAR_PLANE_TOL_REL,  # 1e-6
    vertex_match_tol_relative: float = VERTEX_MATCH_TOL_REL,      # 1e-6
    # Coverage gate for MismatchedContactRegion warnings.
    coverage_epsilon: float = COVERAGE_EPSILON,                   # 1e-3
) -> OverlapReport: ...
```

**Module-level tolerance constants** (defined in `raysim.geom.overlap`,
re-exported from `raysim.geom`):

| Constant | Default | Meaning |
|---|---|---|
| `COPLANAR_NORMAL_TOL_RAD` | `1e-6` | Max angular deviation between two face normals to be considered parallel-or-antiparallel |
| `COPLANAR_PLANE_TOL_REL` | `1e-6` | Max signed-distance/bbox-diag for two coplanar faces to be on the same plane |
| `VERTEX_MATCH_TOL_REL` | `1e-6` | Per-axis rounding bin (× bbox-diag) for the vertex-match hash |
| `COVERAGE_EPSILON` | `1e-3` | If matched-area / shared-area falls below `1.0 - COVERAGE_EPSILON`, fire a `MismatchedContactRegion` warning |
| `ZERO_VOLUME_REL` | `1e-9` | Below this fraction of `min(V_a, V_b)`, the intersection is treated as zero (no penetration) |
| `NESTED_VOLUME_REL` | `1e-6` | If `V_int / min(V_a, V_b)` is within this of 1.0, classify as `accepted_nested` |

### 6. `raysim.geom.pipeline` (orchestrator)

**File**: `src/raysim/geom/pipeline.py` (new)

**What it does**
- Glues B1.1→B1.5 into a single call:

  ```
  load_step → iter_leaves → tessellate (per leaf)
            → heal_assembly → validate_watertightness → diagnose_overlaps
            → return ValidatedAssembly
  ```

- **`ValidatedAssembly`** carries the healed solids, per-solid normalized
  triangles, the watertightness and overlap reports, and the source
  STEP path + tessellation parameters (for provenance hashing in B2.5).
- The pipeline does **not** raise on watertightness or interference
  failures — it returns the reports and lets the caller (the CLI /
  Stage B UI) decide. The actual gating happens in B1.6's adapter,
  which by default raises on:
  - any non-watertight shell (override:
    `accept_watertightness_failures=True`);
  - `interference_fail` solid pairs (override: both
    `accept_warnings=True` *and* `accept_interference_fail=True`,
    deliberately a two-key opt-in because the bias is large);
  - `interference_warning` solid pairs (override: `accept_warnings=True`);
  - `MismatchedContactRegion` warnings (override: `accept_warnings=True`).
  All overrides used are recorded on the returned `ValidatedAssembly`
  so reports surface them. `accepted_nested` and `contact_only` solid-
  pair statuses pass through silently *unless* a `MismatchedContactRegion`
  warning is recorded for the same pair, in which case the warning
  fires its own gate independent of the pair's status.

**Public surface**
- `class ValidationOverrides` (frozen dataclass): three booleans
  (`accept_warnings`, `accept_interference_fail`,
  `accept_watertightness_failures`) recording which gates the caller
  bypassed. Pipeline returns it all-False; adapter replaces it with
  the actual flags supplied to `build_scene_from_step`.
- `class ValidatedAssembly`: includes `overrides_used: ValidationOverrides`
  alongside the reports — see §Type Definitions.
- `def build_assembly_from_step(step_path, *, linear_mm=0.1,
  angular_rad=0.5) -> ValidatedAssembly`. Returns
  `ValidatedAssembly` with `overrides_used = ValidationOverrides()`
  (all defaults). Mutation happens in the adapter.

### 7. `raysim.geom.adapter` (B1.6)

**File**: `src/raysim/geom/adapter.py` (new)

**What it does**

Per the user's chosen approach: **STL-export-then-load** — write per-leaf-
solid STLs to a tempdir, then call `raysim.ray.scene.load_scene_from_directory`.
A small, additive change to `raysim.ray.scene` (a new `tied_groups=`
parameter) lets us pass B1.5's richer STEP-derived coincident-face
groupings into the engine without re-detecting in the limited Stage A
detector.

- One STL per leaf solid, named `<solid_id>.stl` (binary STL).
- Generate a `MaterialAssignment[]` JSON listing
  `{solid_id, material_group_id}` for every leaf, derived from the
  user-supplied assignment table (CLI `--assignments` flag in Stage A;
  Stage B UI later).
- **Deterministic STL output**: implemented as an **in-house binary-STL
  writer** (~30 LOC: 80-byte header, `uint32 n_triangles`, then per
  triangle 12 little-endian `float32`s + 2-byte `attribute_byte_count`
  set to 0). Trimesh's `mesh.export(format="stl")` is *not* used
  because it deduplicates and re-indexes vertices internally, which
  defeats byte-stability. The writer emits triangles in the order the
  adapter supplies, after the adapter has lex-sorted triangles by
  `(v0_rounded, v1_rounded, v2_rounded)` with vertex coords rounded
  to 6 decimals (micron — finer than any physical resolution that
  matters for sector-analysis ∑ρL on mm-scale geometry; coarser than
  the previous 12-decimal rounding that was overkill). The adapter
  records the **stable triangle index map** per solid
  (`triangle_index_map: NDArray[int64]` where
  `triangle_index_map[flat_tessellation_index] = export_index`) so
  B1.5's per-solid triangle indices can be translated into post-export
  `(geom_id, prim_id)` pairs.
- **Tied-group handoff**: after `load_scene_from_directory` returns a
  `BuiltScene`, the adapter overwrites
  `BuiltScene.tied_group_id_per_geom` and
  `BuiltScene.tied_group_members` with the STEP-derived size-2 pairs
  from `OverlapReport.all_tied_triangle_pairs()` (translated via the
  STL export's `triangle_index_map`). For topology-shared contacts —
  which is what B1's scope covers — the resulting groups are
  bit-equivalent to what Stage A's vertex-set hash would produce on
  the same triangulation. The handoff exists because (a) the geometry
  pipeline already has authoritative pair information from OCCT face
  topology and shouldn't re-detect post-export, and (b) it's the
  natural integration point when a future phase adds mesh-Boolean
  re-triangulation for mismatched contacts.
- **Watertightness gate**: raises `GeomValidationError` if any shell in
  `WatertightnessReport.failed_shells()` (the new accessor on the
  report) does not pass, unless `accept_watertightness_failures=True`
  is supplied. Stack traversal assumes closed, outward-oriented shells;
  an unpaired edge or same-orientation pair leaves rays with non-empty
  stacks at miss = wrong ∑ρL. The override exists for partial overrides
  (a known-bad shell on a non-load-bearing piece the user has decided
  to accept anyway) and is recorded in the returned `ValidatedAssembly`
  so reports can surface it.
- **Overlap gate**: raises `GeomValidationError` for **any** of the
  following unless `accept_warnings=True` is supplied:
  - `OverlapReport.failed()` non-empty (`interference_fail`) — `accept_warnings`
    does **not** unblock these; they always require a CAD fix or, as a last
    resort, both `accept_warnings=True` *and* a separate
    `accept_interference_fail=True` flag (kept distinct because the bias
    is large enough that opting in should be deliberate per pair).
  - `OverlapReport.warnings()` non-empty (`interference_warning`).
  - `OverlapReport.mismatched_contacts` non-empty
    (`MismatchedContactRegion`). A mismatched coplanar contact means a
    ray traversing the contact will hit one side, advance by `eps`, and
    miss the other → wrong stack → wrong ∑ρL. Critical to gate.
  Note: a `contact_only` solid pair *can* coexist with a
  `MismatchedContactRegion` warning (they are independent dimensions —
  classification by volume vs. tessellation alignment per coplanar
  region). The `contact_only` status itself passes silently, but the
  parallel mismatched-contact entry on `OverlapReport.mismatched_contacts`
  triggers the gate. `accepted_nested` and `contact_only`
  *without* a parallel mismatched entry pass through silently.
- Public function: `def build_scene_from_step(step_path, *, materials,
  assignments, linear_mm=0.1, angular_rad=0.5,
  accept_warnings: bool = False,
  accept_interference_fail: bool = False,
  accept_watertightness_failures: bool = False) ->
  tuple[BuiltScene, ValidatedAssembly]`. The override flags are
  recorded in the returned `ValidatedAssembly` so reports surface the
  exact set of overrides used.
- Public function: `def export_assembly_to_stl(assembly: ValidatedAssembly,
  out_dir: Path) -> tuple[ExportedSolid, ...]`. `ExportedSolid` carries
  `solid_id`, `path`, and the `triangle_index_map` so callers can wire
  external diagnostics back to per-solid triangles.

## Technical Considerations

### Pattern usage (from ARCHI §5)
- **Detector-centric** ✅ — B1 doesn't touch detectors; engine input
  shape is preserved.
- **Material truth is governed** ✅ — assignments still flow through
  `MaterialAssignment[]`; B1 doesn't touch the material library or
  invent new sources of truth. Per-part STEP material tags land with
  B2.2.
- **Determinism** — see below; STEP→STL ordering is the new surface
  that needs explicit pinning.

### Determinism impact (ARCHI §14)
- New input: STEP file. Will hash into the project file (`.raysim`)
  via B2.5; **not yet** in `Provenance` of `run.json` because the CLI
  Stage A path still consumes STL. **No `Provenance` schema bump in
  B1**. When B2.5 lands, `geometry_hash` will hash the STEP bytes plus
  the tessellation parameters (linear/angular deflection) so equivalent
  STEPs at different deflections produce different hashes.
- Ordered reductions inside the new modules:
  - Leaf walk order: depth-first on the compound's child enumeration
    order (deterministic from OCCT for a given STEP).
  - Triangle ordering inside an STL: lex-sort by
    `(rounded_v0, rounded_v1, rounded_v2)` with 6-decimal (micron)
    rounding — ensures byte-identical STL on two consecutive exports
    of the same healed assembly. Written by the adapter's in-house
    binary-STL writer (no trimesh re-indexing).
  - Solid pair iteration in `diagnose_overlaps`: lex on
    `(solid_id_a, solid_id_b)` with `solid_id_a < solid_id_b`.
- Canonical JSON for the assignments file emitted by the adapter:
  uses `raysim.proj.canonical_json` already.

### Numerical precision (ARCHI §15)
- All geometry math in `float64`. Triangle vertex coordinates retained
  in `float64` through the pipeline; the `float32` boundary at Embree
  is unchanged from Stage A.
- Probe-ray classification in B1.3 uses `float64` `dot(direction,
  triangle_normal)`; tolerance for the dot-product zero band is
  `1e-9` (treated as "tangent — re-pick probe direction").
- Overlap intersection volume from BOPAlgo is OCCT-native double
  precision.
- A.7 1e-5 hard gate is unaffected by B1 — the gate runs on Stage A's
  STL path, and the B1.6 STL export is byte-comparable to the analytic
  fixtures (volume match within 0.5% per `MVP_STEPS §B1.6` — looser
  than the ray-test gate because tessellation deflection is the
  controlling error).

### Material physics scope
- Unchanged. `density_g_cm3` only. `z_eff`, composition still metadata.
  B1 doesn't touch `proj.schema`.

### Coincident-face / tied-batch (ARCHI §11)
- B1.5's tied pairs are **size-2 vertex-match pairs** within the
  shared planar region of two coplanar OCCT faces — same shape and
  semantics as Stage A's existing vertex-set detector, just sourced
  from the geometry pipeline rather than re-detected at scene build.
- The tracer's tied-batch contract requires every member of a group
  to be at the same 3D point within float-noise. B1's tied groups
  honor that (size-2 only). B1.5 does **not** lump every triangle of
  a contact face into one mega-group — that would violate the
  contract and produce stack mismatches.
- **Why the handoff exists.** The Stage A vertex-set detector is
  scoped to legacy STL inputs and re-detects at scene build. For
  STEP inputs we already know the authoritative pairings from OCCT
  face topology; we pass them through `PreBuiltTiedGroups` so the
  scene loader doesn't redundantly hash and so the integration point
  is in place when a future phase adds mesh-Boolean re-triangulation
  for the mismatched-tessellation case (see B1.5 scope decision).
- Both detectors emit the **same data shape** (`(geom_id, prim_id) →
  tied_group_id` + `tied_group_members`), so the traversal in
  `raysim.ray.tracer` is unchanged. Tie-batch sort order
  `(geom_id, prim_id)` ascending preserved.
- The Stage A vertex-set detector is *not* deprecated — it remains
  the authoritative path for legacy STL inputs (e.g.
  `tests/integration/test_phase_a_acceptance.py`).
- **Mismatched-tessellation contacts are not converted to tied
  groups in B1.** They are surfaced as `MismatchedContactRegion`
  warnings and gated by the adapter's `accept_warnings` flag.
  Mesh-Boolean re-triangulation is a follow-up.

### Backend optionality
- All B1 modules guard `import OCC.Core.*` at function body or behind
  `if TYPE_CHECKING`. The package `raysim.geom` itself is import-safe
  even without OCC (defines its dataclasses and raises a clear
  `RuntimeError` only when a function actually needs OCC).
- New pytest marker `needs_occt` registered in `pyproject.toml`.
- `[geom]` extra in `pyproject.toml` already exists as a documentation-
  only stub (per phase-0); B1 keeps it that way (no PyPI distribution
  for `pythonocc-core`).

### Schema versioning
- No `proj.schema` or `env.schema` change in B1. `RunResult.Provenance`
  unchanged. **No schema_version bump.** B2.5 introduces the `.raysim`
  project file and is where geometry-hash policy lands.

### Edge cases
- **Empty STEP** (no solids): `load_step` raises `ValueError`.
- **Degenerate solids** (zero volume): tessellation skips them, logged
  as `warn`; pipeline continues.
- **Healer regression** (healer removes more than it should — possible
  on pathological STEPs): we run the healer once. The "twice produces
  no further changes" check is in tests, not at runtime — runtime cost
  isn't justified for the safety it gives.
- **Probe-ray tangent grazing**: the +X probe direction is hard-coded;
  if a tangent is detected (dot < 1e-9 over multiple consecutive hits),
  pick a fallback direction (+Y, then +Z). Logged.
- **`interference_fail`** outside the user's intent: the adapter raises
  by default. The two-key opt-in (`accept_warnings=True` *and*
  `accept_interference_fail=True`) is the only way to proceed without
  a CAD fix; both flags are recorded on the returned
  `ValidatedAssembly.overrides_used` so reports surface the choice.
- **OCCT boolean failure** (`BRepAlgoAPI_Common` or `BOPAlgo_BOP` returns
  an error / `IsDone() == False`): treat the pair as undetermined —
  log at warning level, record as a `MismatchedContactRegion`-style
  diagnostic in `OverlapReport.boolean_failures` (a parallel tuple
  field), gate it under `accept_warnings=True`. The pair's volume-based
  classification is recorded as `interference_warning` with
  `intersection_volume_mm3 = nan` so the user is told something
  uncertain happened.
- **Multi-cavity solid** with two cavities sharing a face (rare,
  pathological): handled by the per-cavity probe-ray test — each
  cavity is normalized independently.

```python
    solids: tuple[HealedSolid, ...]
    watertightness: WatertightnessReport
    overlaps: OverlapReport
    overrides_used: ValidationOverrides
        # Defaults to all-False when build_assembly_from_step() returns
        # directly. The adapter mutates a copy with the actual flags
        # used and replaces the field on the returned assembly.

# adapter.py
class GeomValidationError(RuntimeError):
    """Raised by build_scene_from_step when watertightness or
    interference gates trip without an explicit override."""

class ExportedSolid:
    solid_id: str
    path: Path
    triangle_index_map: NDArray[int64]  # tessellation_idx -> export_idx

# raysim.ray.scene (additive)
class PreBuiltTiedGroups:
    tied_group_id_per_geom: tuple[NDArray[np.int32], ...]
    tied_group_members: Mapping[int, tuple[tuple[int, int], ...]]
```

## Performance & Cost Impact

- B1 is **off the engine hot path**. Tessellation and healing run once
  per scene load, milliseconds-to-seconds even on the larger benchmark
  fixtures. No `pytest-benchmark` markers added in B1; performance
  budget belongs to B5.4.
- The `O(N²)` solid-pair AABB scan in `diagnose_overlaps` is fine for
  the benchmark assemblies (≤ ~50 solids in the custom test article).
  When real STEPs land in B2.x with hundreds of solids, revisit with
  an AABB-tree pre-filter — out of B1 scope.
- STL export tempdir: cleaned up in a `finally` block so failed runs
  don't leak files. Default `TemporaryDirectory()` placement (system
  tmp) — not configurable in B1.

## Backward Compatibility

- Stage A's `raysim run` CLI still consumes STLs. B1 adds the STEP
  path *alongside*. No CLI flag changes, no deprecations. The STEP-
  enabled CLI (likely `raysim run --scene-step <path>` or analogous)
  lands when B2.5 arrives — out of B1 scope.
- `BuiltScene` shape unchanged.
- `Provenance` schema unchanged (still v2 — `assignments_hash` already
  in place).
- Default `uv run pytest` still passes without `pythonocc-core` —
  B1 tests skipped via `importorskip`.
